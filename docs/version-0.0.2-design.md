# Версия 0.0.2: ISEA3H-гексасфера и процедурная визуализация

Issue #7 просит начать с трёхмерного полигонального шара из гексагонов,
который можно вращать и масштабировать, но сразу закладывает ограничения
будущей игры: планета радиусом примерно в 10 раз меньше Земли, гексы около
10 метров, память сервера около 1 ГБ, синхронизация событиями, а не полной
геометрией, и дальнейшие слои городов, биомов, дорог, энергетики, климата,
водных и минеральных ресурсов.

## ISEA3H как логическая сетка

`game1.hex_sphere.Isea3hGridSpec` хранит параметры логической сетки:

- проекция: ISEA, то есть Icosahedral Snyder Equal Area;
- topology: hexagonal;
- aperture: 3;
- число ячеек на уровне `resolution`: `10 * 3**resolution + 2`;
- радиус по умолчанию: `637100 м`, то есть 1/10 радиуса Земли.

Для цели `10 м` на такой планете метод
`Isea3hGridSpec.for_target_hex_edge_m(10)` выбирает разрешение, где средняя
площадь ячейки уже не превышает площадь правильного 10-метрового гекса.
Это даёт десятки миллиардов логических ячеек, поэтому код не создаёт список
всех гексов. Вместо этого он оценивает объём состояния и явно показывает,
что такой уровень должен грузиться чанками.

Этот выбор следует обычной DGGS-архитектуре: сначала задаются базовый
икосаэдр, ориентация, проекция ISEA и aperture/refinement, а затем к ячейкам
привязываются данные. См. PROJ про `+proj=isea` и описание DGGRID/STCL:

- https://proj.org/en/stable/operations/projections/isea.html
- https://www.discreteglobalgrids.org/information/

## Процедурный меш

`build_hex_sphere_mesh()` строит визуальный меш как dual-сетку
подразделённого икосаэдра:

- вершины исходного икосаэдра становятся 12 пентагонами;
- остальные вершины дают гексагональные ячейки;
- границы ячеек хранятся в единичных координатах сферы;
- биомы и высоты генерируются детерминированно от центра ячейки;
- клетки группируются в 20 face-chunks для быстрой выборки.

Для чётных уровней ISEA3H число ячеек визуального dual-меша совпадает с
логическим количеством. Например, `resolution=4` использует `frequency=9`
и создаёт `812` клеток: `10 * 3**4 + 2`.

Нечётные уровни ISEA3H требуют повёрнутого Goldberg-transform. Логическая
сетка уже поддерживает такие уровни, но текущий лёгкий визуализатор точно
совмещает render-cell-count с logical-cell-count только для чётных уровней.
Это достаточный первый слой для просмотра, чанков, индексов и будущего
перехода к более точной DGGS-геометрии.

## Чанки и пространственный индекс

`CellSpatialIndex` сначала отбирает face-chunks по углу между камерой и
центром чанка, а уже затем фильтрует ячейки по угловому радиусу. Это не
полный серверный spatial index, но публичный интерфейс уже соответствует
нужной модели: искать рядом с камерой или объектом без полного перебора
всей планеты.

Для будущих 10-метровых гексов сервер должен хранить только активные чанки,
видимые чанки и агрегаты по более грубым уровням. Состояние каждой логической
ячейки нельзя держать в памяти в виде Python-объекта.

## События вместо полигонов

`CellEvent` сериализуется так:

```json
{
  "tick": 12,
  "actor": "player-a",
  "cell": "#2000000000000",
  "action": "start_build",
  "payload": {"building": "solar_panel"}
}
```

В событии нет `vertices`, `boundary` или полной геометрии. Клиент и сервер
знают сетку и восстанавливают положение по id ячейки.

## Просмотрщик

`examples/run_hex_sphere_viewer.py` генерирует
`examples/hex_sphere_viewer.html`. Файл самодостаточный: внутри лежит JSON
чанков и Canvas-рендер без внешних JavaScript-зависимостей.

```bash
python examples/run_hex_sphere_viewer.py
```

Открытый HTML показывает полигональную сферу, поддерживает вращение указателем,
масштабирование колесом или кнопками и выбор ячейки для просмотра её id, биома
и высоты.
