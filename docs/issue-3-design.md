# Версия 0.01: планетарный слой по issue #3

Issue #3 ([ссылка](https://github.com/PavelChurkin/resource-based_economy_strategy/issues/3))
описывает стратегию в духе Timberborn / Rimworld / Oxygen Not Included с
ключевым правилом: «Никаких внутриигровых валют — только ресурсы».
Issue #3 пересекается с issue #1, но делает дополнительный акцент на
полигональной сфере-планете, климате с циклонами, рельефе под ГЭС, дереве
технологий из внешнего репозитория `techno2.json` и масштабировании
времени с пропуском недель/месяцев/лет.

Этот PR добавляет проверяемые модули именно под эти акценты, не пересекаясь
с ядром симуляции из PR #2 (issue #1).

## Выбор пакета `game1/`

Имя выбрано чтобы отразить заголовок issue «Мои предположения по игре1»
и не конфликтовать с пакетом `resource_based_economy_strategy/` из PR #2.
Слои планеты, климата и времени намеренно живут в собственном пакете и
зависят только от стандартной библиотеки. Когда PR #2 будет смерджен,
ядро симуляции сможет использовать `game1.TimeController` для управления
тиками и `game1.Planet` для координат поселений на сфере.

## Полигональная планета

`game1.planet.Planet` — список тайлов на сетке широт/долгот. Каждый тайл
несёт `Terrain` с биомом, высотой рельефа и параметрами реки, плюс словарь
ресурсов, привязанных к биому. Метод `hydropower_potential_kw()` оценивает
мощность ГЭС по формуле `P = ρ g Q h η`, что моделирует требование issue:
«чем больше перепад высот у реки — тем больше энергии».

Функция `build_demo_planet()` возвращает детерминированную тестовую
планету с полярными шапками, тундрой, умеренной зоной и тропиками. Сетка
с шагом 30°×60° даёт 42 тайла — дешево для тестов и юнит-сценариев. API
сохраняется при переходе на более плотную тесселяцию (например,
икосаэдрическую), как и предполагается развитием версий.

## Климат: циклоны и антициклоны

`game1.climate.Climate.weather_for(...)` возвращает детерминированный
снимок погоды для конкретного дня и тайла. Состояние давления переключается
между `cyclone` → `transition` → `anticyclone` по периодическому циклу,
причём:

- циклон режет солнечную генерацию (×0.45) и поднимает осадки до 14 мм/день,
  бонус к гидрогенерации (×1.25),
- антициклон даёт солнечный буст (×1.1) и сухость (1.5 мм),
- переход между ними даёт прибавку ветра (×1.4 от базового), как и
  описывает issue.

Сезонная температура считается через синус по году с поправкой на широту
и адиабатический градиент 6.5 °C/км по высоте.

## Совместимость с `techno2.json`

`game1.tech_tree` парсит JSON со схемой:

```json
{
  "технологии": [
    {"название": "Колесо", "описание": "...", "условия": []},
    {"название": "Автомобиль", "условия": ["Двигатель внутреннего сгорания", "Бензин"]},
    {"название": "Абразив", "условия": [["Карбид кремния"], ["Алмаз"]]}
  ]
}
```

Плоский список — конъюнкция (AND), список списков — дизъюнкция конъюнкций
(OR-of-AND). Метод `TechTree.unknown_dependencies()` возвращает все
требуемые имена, для которых в дереве нет узла-определения, что удобно
для подсева начального инвентаря и для миграции внешнего дерева.

## Контроллер времени

Issue требует:

- паузу для осознанной игры,
- нормальный режим (10 c за игровой день),
- ускоренный режим (1 c за день),
- пропуск недель / месяцев / лет через экстраполяцию,
- возможность увеличить длительность дня под нагрузку.

`game1.time_control.TimeController`:

- принимает любой объект с методами `tick()` и `daily_summary()` (Protocol),
- использует инжектируемый `clock` (для тестов — фейк, для прод-кода —
  `time.monotonic`),
- метод `step()` запускает столько `tick()`, сколько прошло реального
  времени с момента предыдущего шага, разделённого на текущий
  `seconds_per_day`,
- метод `fast_forward(days)` НЕ вызывает `tick()`, а умножает дневную
  сводку на число дней — это и есть «вычисление статистики и её
  экстраполяция на будущее» из формулировки issue,
- хелперы `skip_weeks` / `skip_months` / `skip_years` оборачивают
  `fast_forward` с понятной семантикой.

Интерфейс контроллера специально не зависит от модели поселения: тот же
контроллер позднее обернёт `Settlement` из PR #2.

## Что осталось за рамками

- Полная игровая петля (постройки, рецепты, потребности людей, медицина,
  износ роботов) — она уже разрабатывается в PR #2.
- Серверный цикл и многопользовательский режим, упомянутый в issue.
- Импорт реального `techno2.json` — модуль умеет его читать, но конкретный
  снапшот не заводится в репозиторий до согласования источника.
- 3D-визуализация полигональной сферы — это слой клиента, а не симуляции.
